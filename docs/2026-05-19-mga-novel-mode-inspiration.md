# 灵感：漫画翻译也能顺手覆盖小说

日期：2026-05-19

## 这段对话里冒出来的想法

`manga-translator-agent` 不一定只服务漫画。它本质上可以被看成一条统一的本地化管线：

- 漫画：`图像 -> 文本 -> 翻译 -> 回写`
- 小说：`文本 -> 翻译 -> 回写`

小说场景只是把最前面的“从图片里抠文字”这一步省掉了，后面的翻译、术语、角色语气、文化适配、QA、输出格式，仍然可以复用同一套核心能力。

## 由此得到的产品直觉

项目边界不该被“漫画”这个输入形态锁死，而应该被“把内容稳定地翻成另一种可读形式”定义。

所以更合适的抽象是：

- 视觉输入层：处理漫画图片、气泡顺序、说话人归属
- 文本输入层：处理小说、EPUB、TXT、PDF
- 共享翻译层：术语、角色档案、文化适配、QA、排版回写

## 这条灵感的意义

漫画只是最难的输入形态，不是系统边界。
如果把它抽象成“多模态文本本地化引擎”，小说模式就只是另一种输入分支，而不是另起一个项目。
